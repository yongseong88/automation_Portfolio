# 지침서: 테스트 실패 디버깅 가이드 (DEBUG_CODE.md)

## 🔍 1. 디버깅 필수 분석 단계
테스트 실패 에러 메시지나 로그를 받으면 클로드는 다음 순서대로 분석을 진행해야 합니다.
1. **에러 타입 식별**: 타임아웃(Timeout), 요소 없음(NoSuchElementException/TimeoutError), 검증 실패(AssertionError) 구별
2. **스크린샷/로그 확인**: 실패 시점의 웹/앱 스크린샷 및 API Response Body 로그를 대조
3. **환경 요인 점검**: 대상 서버(Staging 등)가 일시적으로 다운되었거나 네트워크 지연이 있는지 확인

## 🛠️ 2. 에러 유형별 해결 체크리스트
- **Element Not Found (요소를 찾을 수 없음)**:
  - 웹 애플리케이션의 DOM 변경 여부 확인 및 `data-testid` 재검증
  - 앱 화면의 로딩 속도로 인한 문제일 경우 `WebDriverWait`를 이용한 명시적 대기(Explicit Wait) 시간 상향 조정
- **Assertion Error (검증 실패)**:
  - 실제 값(Actual)과 기대 값(Expected)의 데이터 타입(String vs Int 등) 일치 여부 확인
  - API 응답 구조가 사전에 정의된 스펙과 달라졌는지 파악
- **Appium 세션 오류**:
  - Appium 서버 로컬 프로세스가 죽었는지, 혹은 에뮬레이터/시뮬레이터 기기 연결이 끊겼는지 점검하도록 가이드

## 🔄 3. 디버깅 후 수정 코드 제안 규칙
- 단순히 "코드의 타임아웃을 늘리세요"라고 하지 말고, **원인이 해결된 구체적인 리팩토링 코드**를 제시하세요.
- 요소 선택자가 변경되었다면 POM 페이지 객체 파일(`pages/`)의 변수만 깔끔하게 수정하는 방식으로 코드를 제안하세요.
